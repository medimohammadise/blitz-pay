# Spring Data JPA Best Practices

Authoritative reference for JPA and persistence conventions in this project.
See `CONTRIBUTING.md` for the link to this document.

---

## Current State

This project uses **Hibernate with `ddl-auto: update`** and PostgreSQL 16.
There are currently no JPA entities or Spring Data repositories. When entities and
repositories are added, follow the patterns in this document.

For schema migration strategy (moving away from `ddl-auto: update`), see
`reference/liquibase-best-practices.md`.

---

## 1. Entity Classes — Use Regular Classes, Not `data class`

**Rule:** Never annotate a Kotlin `data class` with `@Entity`.

**Why:**
- JPA proxies require subclassing — `data class` is effectively final in proxy scenarios
- `equals`/`hashCode` generated by `data class` are based on all fields including `id`,
  which breaks JPA identity semantics (a new entity before save has no id)
- `copy()` bypasses JPA change tracking, leading to silent merge issues
- Hibernate requires a no-arg constructor — `data class` makes this awkward

**Do this:**

```kotlin
@Entity
@Table(name = "payment_requests")
class PaymentRequest(
    @Id
    val id: UUID = UUID.randomUUID(),

    @Column(nullable = false)
    val orderId: String,

    @Column(nullable = false)
    val amountMinorUnits: Long,

    @Column(nullable = false, length = 3)
    val currency: String,

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    var status: PaymentStatus = PaymentStatus.PENDING,

    @Column(nullable = false, updatable = false)
    val createdAt: Instant = Instant.now()
) {
    override fun equals(other: Any?): Boolean {
        if (this === other) return true
        if (other !is PaymentRequest) return false
        return id == other.id
    }

    override fun hashCode(): Int = id.hashCode()
}
```

**Not this:**

```kotlin
// Wrong — data class for JPA entity
@Entity
data class PaymentRequest(
    @Id val id: UUID = UUID.randomUUID(),
    val orderId: String,
    ...
)
```

## 2. ID Generation

Prefer `UUID` assigned in Kotlin (not `IDENTITY` / `SEQUENCE` auto-generation) for
entities that participate in events before being persisted.

```kotlin
@Id
val id: UUID = UUID.randomUUID()
```

**Why:** The entity ID is known immediately at creation time. You can publish it in an event
before the transaction commits, without a round-trip to the database to get a generated key.

If using database-generated IDs is preferred for a specific entity, use `SEQUENCE` —
never `AUTO` (which falls back to a table-based strategy on some dialects).

```kotlin
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "payment_seq")
@SequenceGenerator(name = "payment_seq", sequenceName = "payment_request_seq", allocationSize = 50)
val id: Long = 0
```

## 3. Spring Data Repositories

Always extend `JpaRepository` or `CrudRepository`. Keep repositories inside their
module — never reference another module's repository directly.

```kotlin
// invoice/internal/InvoiceRepository.kt
interface InvoiceRepository : JpaRepository<Invoice, UUID> {

    fun findByStatusAndCreatedAtBefore(
        status: InvoiceStatus,
        cutoff: Instant
    ): List<Invoice>

    @Query("SELECT i FROM Invoice i WHERE i.buyerVatId = :vatId ORDER BY i.createdAt DESC")
    fun findRecentByBuyer(@Param("vatId") vatId: String, pageable: Pageable): Page<Invoice>
}
```

**Rules:**
- Repository interfaces live in the owning module's `internal` package
- Never expose a repository through a module's public API — expose a service instead
- Derived query methods are fine for simple cases; use `@Query` for anything with JOINs
  or ordering to keep the intent explicit

## 4. `open-in-view: false`

This project already has `open-in-view: false` in `application.yml`. Keep it that way.

**Why:** The Open Session in View anti-pattern keeps the JPA session open through the
HTTP response rendering phase, leading to N+1 query issues and misleading transaction
boundaries. With WebFlux (reactive), it's not supported anyway.

## 5. Transactions

Use `@Transactional` on service methods, not on repositories or controllers.

```kotlin
@Service
class InvoiceService(
    private val invoiceRepository: InvoiceRepository,
    private val eventPublisher: ApplicationEventPublisher
) {
    @Transactional
    fun createInvoice(data: InvoiceData): Invoice {
        val invoice = Invoice(/* ... */)
        invoiceRepository.save(invoice)
        eventPublisher.publishEvent(InvoiceCreated(invoice.id))  // published after commit
        return invoice
    }

    @Transactional(readOnly = true)
    fun findById(id: UUID): Invoice =
        invoiceRepository.findById(id).orElseThrow { InvoiceNotFoundException(id) }
}
```

**Rules:**
- `readOnly = true` on query-only methods — allows Hibernate optimisations and
  signals intent clearly
- Never put `@Transactional` on a `private` method — Spring proxies won't intercept it
- Keep transactions short — don't call external HTTP services inside a transaction
- Listeners that do DB work should use `@ApplicationModuleListener` (runs after commit)
  rather than `@EventListener` (runs inside the publishing transaction)

## 6. Column Mapping Conventions

```kotlin
@Entity
@Table(name = "invoices")   // explicit snake_case table name
class Invoice(

    @Id
    val id: UUID = UUID.randomUUID(),

    @Column(name = "invoice_number", nullable = false, unique = true)
    val invoiceNumber: String,

    @Column(name = "issue_date", nullable = false)
    val issueDate: LocalDate,

    @Column(name = "status", nullable = false, length = 20)
    @Enumerated(EnumType.STRING)        // never EnumType.ORDINAL
    var status: InvoiceStatus,

    @Column(name = "created_at", nullable = false, updatable = false)
    val createdAt: Instant = Instant.now(),

    @Column(name = "amount_cents", nullable = false)
    val amountCents: Long                // store money as minor units (cents), never BigDecimal in DB
)
```

**Rules:**
- Always use `@Table(name = "…")` and `@Column(name = "…")` explicitly — don't rely on
  Hibernate's naming strategy to guess snake_case from camelCase
- Always use `EnumType.STRING` — ordinal values break if enum members are reordered
- Store money as `Long` minor units in the database, convert to `BigDecimal` in the domain layer
- Mark immutable columns `updatable = false`

## 7. Kotlin + JPA Compatibility (`@Entity` open classes)

Hibernate requires entity classes and their methods/fields to be `open` for proxy generation.
Add the `kotlin-allopen` plugin in Gradle to avoid annotating every class manually.

```kotlin
// build.gradle.kts
plugins {
    kotlin("plugin.jpa") version "…"         // adds no-arg constructor to @Entity classes
    kotlin("plugin.allopen") version "…"     // makes @Entity classes open automatically
}

allOpen {
    annotation("jakarta.persistence.Entity")
    annotation("jakarta.persistence.MappedSuperclass")
    annotation("jakarta.persistence.Embeddable")
}
```

Without this, Hibernate lazy loading will fail silently.
